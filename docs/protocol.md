# Sentinel 지표 전송 프로토콜 v1

에이전트(C++) → 서버(Kotlin) 방향으로 지표를 실어 보내는 자체 TCP 프로토콜.

## 왜 자체 프로토콜인가 — TCP의 함정

TCP는 "메시지" 단위가 아니라 **바이트 흐름(byte stream)**이다. 송신 측이
`[메시지A][메시지B]`를 한 번에 보내도, 수신 측은 `[메시지A][메시]` … `[지B]`
처럼 임의로 쪼개져 도착할 수 있다(반대로 여러 메시지가 붙어 오기도 한다).

따라서 "여기서부터 여기까지가 한 메시지"라는 **경계(framing)**를 애플리케이션이
직접 정의해야 한다. 이 프로토콜은 가장 흔한 방식인 **길이 접두(length-prefix)**를 쓴다.

> 비유: 택배 상자에 "이 상자 내용물은 몇 kg"이라고 먼저 적어두면, 받는 사람은
> 저울로 그만큼만 덜어내면 된다. 길이를 모르면 어디까지가 한 상자인지 알 수 없다.

## 프레임 구조

한 프레임 = **4바이트 길이 헤더 + JSON 본문**.

```
+-------------------+--------------------------------+
| length (4 bytes)  | JSON payload (length 바이트)   |
| uint32, big-endian| UTF-8                          |
+-------------------+--------------------------------+
```

- `length`: 뒤따르는 JSON 본문의 **바이트 수**. 부호 없는 32비트 정수.
  네트워크 바이트 순서(**big-endian**)로 보낸다.
  - 왜 big-endian인가: 네트워크의 관례(network byte order). 송·수신 머신의
    CPU 엔디안이 달라도 해석이 어긋나지 않도록 한 쪽으로 통일한다.
    C++는 `htonl()`, Kotlin(JVM)은 기본이 big-endian이라 `readInt()`로 바로 읽힌다.
- `payload`: UTF-8로 인코딩한 JSON 문자열. 아래 스키마를 따른다.
- 한 연결(TCP connection) 위로 프레임을 **연속해서** 보낸다(매 수집 주기마다 1프레임).

### 수신 측 읽기 절차

1. 정확히 4바이트를 읽어 `length`를 얻는다.
2. 정확히 `length`바이트를 읽어 JSON 본문을 얻는다.
3. JSON을 파싱해 처리한다. 1번으로 돌아간다.

"정확히 N바이트"가 핵심이다. 한 번의 read가 N바이트를 다 준다는 보장이 없으므로,
N바이트가 모일 때까지 반복해서 읽어야 한다(`readFully` 계열이 이 일을 해준다).

## JSON 스키마

```json
{
  "host": "agent-01",
  "ts": 1721470000,
  "metrics": [
    { "name": "cpu", "value": 3.72 },
    { "name": "mem", "value": 30.27 }
  ]
}
```

| 필드 | 타입 | 설명 |
|---|---|---|
| `host` | string | 지표를 보낸 호스트 식별자 |
| `ts` | number(int64) | 수집 시각 (Unix epoch 초) |
| `metrics` | array | 지표 목록 |
| `metrics[].name` | string | 지표 이름 (예: `cpu`, `mem`) — 에이전트의 `MetricCollector::Name()` |
| `metrics[].value` | number(double) | 측정값 (0~100 %) |

`metrics`를 배열로 둔 이유: 에이전트에 지표를 추가해도 스키마가 바뀌지 않는다.
(수집기 인터페이스를 배열로 순회하는 구조와 그대로 맞물린다.)

## 한계 / 향후

- v1은 **단방향**(에이전트→서버)이며 서버 응답(ACK)이 없다.
- 인증·암호화 없음(로컬 개발 전제). 운영 시 TLS/토큰은 이후 Phase에서 검토.
- 프레임 최대 크기는 서버에서 64KiB로 제한한다(비정상 length로 인한 메모리 폭주 방어).
