03

핵심 개념 카드

이번에 새로 익힌 개념들. 비유 한 줄과 함께 다시 새겨봅니다.

Counter vs Gauge

측정의 두 종류

주행거리계(누적) vs 속도계(현재). CPU는 누적, 메모리는 현재.

CPU는 누적 시간이라 두 번 읽어 차이를 내야 사용률이 나옵니다(counter). 메모리는 지금 이 순간의 상태라 한 번만 읽으면 됩니다(gauge).

/proc 가상 파일

리눅스 커널의 창

파일처럼 생겼지만 디스크에 없는, 커널이 실시간으로 만들어 보여주는 화면.

/proc/stat은 한 줄에 숫자가 늘어선 형태, /proc/meminfo는 키: 값 쌍. 형식이 달라 파서도 달라집니다.

MemAvailable

착시를 피하는 값

빈 책상 + 책만 치우면 앉을 수 있는 책상까지 세는 것.

리눅스는 남는 메모리를 디스크 캐시로 씁니다. 그 캐시는 필요하면 즉시 반납되므로 "사실상 사용 가능". MemFree만 보면 사용률이 부풀어 보입니다.

RAII

C++ 자원 관리

방을 나가면 불이 자동으로 꺼지는 것 — 자원 정리를 객체 수명에 맡긴다.

ifstream은 스코프를 벗어나면 파일을 자동으로 닫고, TcpSender는 소멸자에서 소켓을 닫습니다. new/delete를 직접 안 씁니다.

인터페이스 & 다형성

확장 가능한 설계

콘센트 규격 — 규격만 맞으면 벽은 뭐가 꽂혔는지 신경 안 쓴다.

순수 가상 함수(= 0)로 "수집기의 계약"을 정하니, main은 개별 타입을 몰라도 Sample()만 부르면 알맞은 구현이 실행됩니다.

가상 소멸자

누수 방지

택배를 실제 주인에게 전달 — 겉포장(부모)만 보고 버리면 안 된다.

부모 포인터로 자식을 delete할 때, 소멸자가 virtual이라야 실제 객체의 소멸자부터 불려 자원이 새지 않습니다.

TCP Framing

프로토콜의 핵심

택배 상자에 "내용물 몇 kg"을 먼저 적기 — 받는 쪽이 어디까지가 한 상자인지 안다.

TCP는 메시지가 아니라 바이트 흐름이라 임의로 쪼개져 옵니다. 4바이트 길이 접두로 "여기부터 여기까지"라는 경계를 직접 만들어야 합니다.

엔디안 / htonl

바이트 순서 통일

서로 다른 나라가 날짜를 한 형식으로 통일해 적는 약속.

머신마다 바이트 순서가 다를 수 있어, 네트워크는 big-endian으로 통일합니다. C++은 htonl로 보내고, JVM은 readInt()로 그대로 읽힙니다.

부분 전송 / 부분 수신

소켓의 함정

양동이로 물 옮기기 — 한 번에 다 안 담길 수 있어 다 될 때까지 반복.

send는 요청한 만큼 다 못 보낼 수 있고, 수신도 쪼개져 옵니다. "정확히 N바이트"가 될 때까지 반복하는 게 정석입니다.

04

막혔던 것 & 해결 — 값진 삽질

port포트 충돌

이 머신은 5432·6379·8080 등을 다른 프로젝트가 이미 쓰고 있었습니다. 기존 걸 건드리지 않고, 접속 포트를 전부 .env로 빼서 5433 · 6380 · 8082 · 9400으로 비켜 잡았습니다. → 교훈: 포트는 코드에 못 박지 말고 환경 변수로.

jvm"Unable to establish loopback connection"

Gradle이 아니라 JVM 문제였습니다. 최소 재현 코드(Selector.open() 한 줄)로 범위를 좁혀 보니, 백신이 Temp 폴더의 소켓만 차단하는 것이 원인. jdk.net.unixdomain.tmpdir을 Temp 밖으로 옮겨 해결. → 교훈: "X가 안 된다" ≠ X의 문제. 레이어를 벗겨 좁혀라.

shellPowerShell의 bash는 WSL

bash dev.sh가 WSL bash(배포판 없음)를 불러 실패했습니다. PowerShell용 dev.ps1을 따로 만들어, 어느 셸에서든 컨테이너 빌드가 되도록 했습니다.

끝단에서 끝단까지 — 실제로 흐른 증거

에이전트가 보낸 값과 서버가 받은 값이 정확히 일치했습니다.

# 에이전트 (컨테이너) sentinel-agent: host=c52ca3140b2c → host.docker.internal:9400 cpu=0.438322% mem=30.2802% -> sent # 서버 (호스트) [c52ca3140b2c] ts=1784525300 cpu=0.438322% mem=30.2802%

4프레임 모두 손실 없이 전달 — 길이 접두 framing, JSON 파싱, 연속 프레임, 연결 종료까지 정상.
